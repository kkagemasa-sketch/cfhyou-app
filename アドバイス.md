## 1. 計算精度の改善（最重要）

### 1-1. 年金計算（今回最も苦労した箇所）

#### 現在の問題点
- **遺族年金の計算が複雑で修正が繰り返し発生した**
  - joinM（加入月数）が退職年齢ではなく死亡年齢を使っていたバグ
  - 廃止された計算方式（2/3・1/2方式）が残っていた
  - 月収をそのまま使うと生涯平均として過大になる

#### 作り直し時の推奨方式

**入力設計（シンプルに）**：
```
老齢基礎年金：            加入年数を入力 → 自動計算
老齢厚生年金：            年金定期便があれば直接入力（必須）
              年金定期便がなければ：標準報酬月額の平均を入力
遺族年金：                自動計算（手入力でオーバーライド可）
```

**遺族厚生年金の正式計算式（2022年改正後）**：
```
65歳未満の妻：
  遺族厚生年金 = 夫の老齢厚生年金 × 3/4

65歳以上の妻（差額方式）：
  遺族厚生年金 = max(夫の老齢厚生年金 × 3/4, 妻の老齢厚生年金)
  ※ただし妻自身の老齢厚生年金を超える部分のみ支給（差額）
  ≒ 妻の老齢基礎年金 + max(夫×3/4, 妻の厚生) + 中高齢寡婦加算
```

**推奨**：
- 年金は自動計算よりも「年金定期便の数値を直接入力させる」方が正確で信頼性が高い
- 自動計算する場合は「あくまで概算」と明示し、年金定期便入力欄をデフォルトにする
- CAREER_FACTOR（0.75）のような補正係数は、ユーザーが意味を理解しにくい → 表示しない

#### 遺族基礎年金の注意点
- 18歳未満の子がいる妻のみ支給（子が18歳になった翌年3月末で停止）
- 子の年齢チェックは年度末（3月）基準だが、簡易計算では整数年齢で十分
- 中高齢寡婦加算：妻40〜64歳で子なし（または子が18歳超）の場合。2024年度：約61.4万円

---

### 1-2. 老齢年金受給額

#### 現在の問題点
- 月収入力 → 0.75補正で計算しているが、補正係数の根拠が説明しにくい
- 「年金定期便がなければ入力できない」という問題を避けるための措置だが、不正確

#### 推奨
```
方式A（推奨）：年金額を直接入力（年金定期便から）
  - 「老齢基礎年金」「老齢厚生年金」を分けて入力
  - 合計が表示される

方式B（簡易）：現在の月収から推計
  - ユーザーに「あくまで目安」と明示
  - 標準報酬月額 × 5.481/1000 × 480ヶ月（40年）× 0.75補正

方式C（詳細）：標準報酬月額の平均を入力
  - 入社〜現在の平均を入力してもらう
```

---

### 1-3. 住宅ローン控除

#### 現在の問題点
- 所得税・住民税の逆算ロジックが複雑でメンテナンスが難しい
- 給与所得控除・社会保険料の計算が簡略化されている

#### 推奨
- 控除額は「手入力（13年分）」を主にする
- 自動計算はボタン1つで「目安を計算」程度に留める
- ローン残高 × 0.7% と税額上限の少ない方が正確な控除額

---

### 1-4. 財形貯蓄・有価証券の運用計算

#### 現在の問題点
- 財形は利息ゼロで計算（実態と乖離）
- 有価証券の複利計算は正しいが、デフォルト5%が強気すぎる可能性

#### 推奨
- 財形貯蓄：利率入力欄を追加（デフォルト0.1%等）
- 有価証券：デフォルト利率を3〜4%程度に変更、またはユーザーが明示的に選択
- インフレ考慮：全体的に実質ベース vs 名目ベースを明示する

## 2. UIの改善提案

### 2-1. 入力パネルの整理

#### 現在の問題点
- セクションが9つあり、何をどこで入力するか分かりにくい
- 動的追加が多く、保険・保険・保険…と縦長になりやすい
- ラベルが小さく（9px〜11px）、特に高齢ユーザーには見にくい

#### 推奨するUI設計
```
【ステップ形式（ウィザード型）】
Step 1: 基本情報（家族構成・年齢）
Step 2: 収入（年収・退職・年金）
Step 3: 住宅（物件・ローン）
Step 4: 生活費（月額・年間）
Step 5: 将来のイベント（教育費・その他支出）
Step 6: 資産（保険・投資）
Step 7: 確認・生成

→ 一画面に全部出すのではなく、ステップを進める形に
```

**別案（タブ型）**：
```
[家族] [収入] [住宅] [生活費] [資産] [万が一]
→ 現在に近い形だが、各タブを整理して項目数を削減
```

#### 具体的な整理提案

| 現在 | 改善後 |
|------|--------|
| ①家族構成 + 保育料 + 教育選択 | 「家族」タブにすっきりまとめる |
| ②自己資産（保険・証券・財形） | 「資産」タブ。保険は種類ごとにリスト表示 |
| ③収入（段階設定） | 「収入」タブ。グラフプレビューで直感的に |
| ⑤住宅（ローン・金利・控除） | 「住宅」タブ。ペアローンは折りたたみ |
| ⑨万が一 | 自動で計算される（生成ボタン不要） |

---

### 2-2. フォントサイズと余白

現在：ラベル 9px、入力値 11px → **視認性が低い**

#### 推奨
- ラベル：最低12px
- 入力値：最低13px
- スマホ対応：16px以上（iOS自動ズーム防止）
- 余白：padding を少し増やし、ごちゃごちゃ感を解消

---

### 2-3. CF表の改善

#### 現在の問題点
- 行数が多く、全体像が把握しにくい
- 「金融資産行を表示/非表示」は機能するが切替が手間

#### 推奨
- **サマリービュー**（デフォルト）：重要行のみ表示（収入・支出・残高の3行程度）
- **詳細ビュー**（展開時）：全行表示
- **カラーコーディング**：赤字年は行全体を薄赤背景
- **ホバー時詳細**：セルにホバーすると内訳をポップアップ表示

---

### 2-4. モバイル対応

現在はスマホ対応しているが、左右分割レイアウトがスマホには不向き。

#### 推奨
- スマホ時は「入力フルスクリーン → 生成 → CF表フルスクリーン」の2画面モード
- または入力部分のみスマホ最適化し、出力は横向きで表示

---

### 2-5. 視覚的なフィードバック強化

- 入力値を変えたときの計算結果の変化を**アニメーション**で表示
- CF表の赤字年に**警告アイコン**
- 総資産がマイナスになる年に**強調表示**

## 3. アーキテクチャの改善

### 3-1. 状態管理の分離

#### 現在の問題点
- `render()` 関数が全計算を担当しており、1,300行以上ある
- DOM から値を直接読み取る設計（`document.getElementById`）がいたるところに

#### 推奨（作り直し時）
```javascript
// 状態オブジェクトを明確に定義
const state = {
  family: { husband: {...}, wife: {...}, children: [] },
  income: { steps: [], pension: {...} },
  housing: { price: 0, loan: {...} },
  livingCost: { monthly: {...}, yearly: {...} },
  assets: { savings: [], lumps: [], securities: [] },
  // ...
};

// 計算は純粋関数で
function calcCF(state) {
  // DOM を読まず、state だけを使って計算
  // 結果を R オブジェクトで返す
  return R;
}

// 表示は別
function renderCF(R) {
  // 計算済みの R を受け取って表示だけする
}
```

**利点**：
- テストが書ける
- 計算ロジックのデバッグが容易
- 複数シナリオの並列計算が簡単

---

### 3-2. 自動保存の改善

#### 現在の問題点
- 2秒デバウンスで保存 → 更新ボタン押下時にデータが消えるバグが発生した
- `saveAndRefresh()` で対処したが、根本的には改善が必要

#### 推奨
```javascript
// ① beforeunload イベントで確実に保存
window.addEventListener('beforeunload', () => {
  saveToIndexedDB(); // 同期的に保存
});

// ② ページ遷移が必要な場合は保存完了を待ってから遷移
async function navigateWithSave() {
  await saveToIndexedDB();
  window.location.href = newUrl;
}
```

---

### 3-3. データ構造の標準化

#### 現在の問題点
- 動的要素のIDが `ins-m-h-1`, `ins-age-h-1` のようにバラバラ
- 保存・復元のコードが複雑

#### 推奨
```javascript
// 配列ベースで管理
state.assets.insuranceSavings = [
  { id: 1, person: 'h', enrollAge: 35, monthly: 3, matAge: 60, matAmt: 500 },
  { id: 2, person: 'w', enrollAge: 30, monthly: 2, matAge: 55, matAmt: 300 },
];

// 表示はこの配列を元に生成
// 保存・読み込みはそのままJSON化
```

---

### 3-4. フレームワークの採用検討

現在は純粋なVanilla JS + DOM操作。作り直し時の選択肢：

| 選択肢 | メリット | デメリット |
|--------|---------|-----------|
| Vanilla JS（現在）| ビルド不要・軽量 | 状態管理が複雑 |
| Vue.js | 学習コスト低・状態管理が楽 | ビルドが必要 |
| React | コンポーネント再利用 | ビルド必要・学習コスト高 |
| Svelte | 軽量・ビルド結果が小さい | エコシステムが小さい |
| Alpine.js | Vanilla JSに近い・軽量 | 複雑な状態管理は難しい |

**個人的推奨**：Alpine.js または Vue.js（CDN版）
- GitHub Pages互換を保ちつつ、状態管理が格段に楽になる
- CDN版なら `<script src="">` だけで使える

## 4. バグになりやすい箇所（注意点）

### 4-1. 年齢計算の落とし穴

```javascript
// ❌ 問題のある計算（今回も発生）
const joinM = (hDeathAge - pHStart) * 12;  // 逝去年齢を使ってしまう

// ✅ 正しい計算
const joinM = Math.max((retAge - pHStart) * 12, 300);  // 退職年齢を使う
```

**ルール**：
- 年金の加入月数計算は常に「退職年齢」を使う
- 逝去年齢は「受給停止」のチェックのみに使う

---

### 4-2. 「奥様の配偶者年金」とご主人様逝去後の重複

現在のコード：
```javascript
R.pW.push((wa>=pWReceive && hAlive) ? ri(pWife) : 0);
// ご主人様逝去後は survPension に統合されるため pW は0
```

**注意**：ご主人様逝去後は
- `pW`（奥様自身の老齢年金）に老齢基礎年金のみを計上
- `survPension` に遺族厚生年金を計上
- 重複計上しないよう注意（現在は survPension に全統合）

---

### 4-3. CF表の表示年数

```javascript
// ❌ 問題が起きた（以前のバージョン）
const totalYrs = Math.max(autoDisp, 60);
// → 50年ローン（delivery + 50 > 60）がCF表に収まらない

// ✅ 現在の正しい計算
const loanEnd = delivery + loanYrs + 1;
const totalYrs = Math.max(autoDisp, 60, loanEnd);
```

**ルール**：表示年数は逝去年齢・ローン終了・最低60年の最大値

---

### 4-4. 動的要素の削除

動的に追加した要素を削除した後、live()を呼ばないと計算が更新されない。
現在はほぼ対処済みだが、作り直し時は「削除 = 状態更新」が確実になるよう設計する。

---

### 4-5. 万が一CF表の状態管理

#### 現在の問題点（今回発生）
- `MR.pS` が存在しないと通常CF表の `R.pS` を参照してしまう
- 万が一CF表のデータは `window._mgStore` にHTMLとして保存している
  → HTMLとして保存は保守性が低い

#### 推奨
- 万が一CF表も Rオブジェクトと同じ構造の MRオブジェクトで持ち、描画は都度行う
- `window._mgStore` のようなHTML文字列保存は避ける

---

### 4-6. ペアローンの住宅ローン控除

ペアローン時は借入限度額が2倍になる想定だが、法律上は各人の借入額ごとに上限が適用される。
現在のコード：
```javascript
const cappedBal = Math.min(remainBal, pairLoanMode ? lctrlLimit*2 : lctrlLimit);
```
これは簡易計算。作り直し時は夫・妻それぞれに別々に計算する方が正確。

---

### 4-7. 修繕積立金のデータ消失バグ（解決済み、再発注意）

**原因**：更新ボタンが `window.location.href = ...` で直接遷移 → 2秒デバウンスの保存が走る前に遷移

**解決策**（実装済み）：
```javascript
async function saveAndRefresh() {
  clearTimeout(_autoSaveTimer);
  await dbPut({...}); // 先に保存
  window.location.href = newUrl; // 保存完了後に遷移
}
```

**作り直し時のルール**：URLパラメータで状態を渡す設計は避ける。すべてIndexedDBに保存し、ページ遷移は不要にする。

## 5. 機能追加の提案

### 5-1. インフレ考慮シミュレーション

現在は全て名目ベース（インフレなし）。
```
インフレ率設定（例：1.5%/年）→ 生活費・教育費を毎年×1.015倍
→ 実質ベースと名目ベースの切り替え表示
```

### 5-2. 複数シナリオの比較グラフ

現在は1シナリオずつしか表示できない。
```
シナリオA（フラット35）vs シナリオB（変動金利）
→ 2シナリオの資産推移を重ねてグラフ表示
```

### 5-3. 年金試算の詳細モード

```
老齢基礎年金の見込み額：
  加入年数を入力 → 満額との比率を表示
  繰上げ受給（60〜64歳）・繰下げ受給（66〜75歳）の比較
```

### 5-4. 教育費の詳細設定

```
現在：公立/私立の2択
改善：
  - 留学オプション（年間200万円増）
  - 予備校・塾費用（中学〜高校）
  - 国際バカロレア（IB）コース
```

### 5-5. 保険料のCF表連動改善

```
現在：積立保険・一時払い保険はCF表の支出に計上される
改善：
  - 保険種類ごとの収支ビューを別タブで表示
  - 保険の死亡保障額を万が一シミュレーションに自動連携
```

### 5-6. 住宅ローン完済後の余裕資金の使い道

```
完済後の余裕資金（月額）を：
A. 貯蓄に回す
B. 繰上返済する
C. 投資に回す
の3パターンを比較表示
```

### 5-7. 住民税非課税世帯チェック

```
年収が一定以下の場合、住民税非課税→
幼保無償化・高等教育無償化の対象判定表示
```

## 6. 開発フローの改善

### 6-1. 計算ロジックのテスト

現在：テストコードなし → バグを手動で確認

**推奨**：
```javascript
// 純粋関数にすれば Jest でテストできる
test('遺族年金の計算（子なし・妻65歳以上）', () => {
  const result = calcSurvPension({
    koseiH: 100, koseiW: 60, kisoW: 81.6, ...
  });
  expect(result).toBe(81.6 + 100*0.75); // 差額方式
});
```

### 6-2. バージョン管理の強化

現在：`?v=11` でキャッシュバスティング

**推奨**：
- `CHANGELOG.md` を書く（何を変更したか記録）
- `git tag v11` でバージョンタグを打つ
- 「この機能を追加したバージョン」を後で追跡できるようにする

### 6-3. 設計原則

作り直し時に守ってほしいこと：

1. **計算関数はDOMを読まない**（`document.getElementById` を使わない）
2. **1つの関数は1つのことをする**（render()が1300行になってはいけない）
3. **定数は constants.js に集中させる**（マジックナンバーをコードに書かない）
4. **コメントを書く**（計算式の根拠・出典を必ず記載）
5. **エラーハンドリング**（入力値が異常でもクラッシュしない）

### 6-4. ドキュメンテーション

年金・遺族年金の計算式は法律改正で変わります。

**推奨**：
- 計算式の出典・根拠を `constants.js` のコメントに書く
- 改正年度ごとの変更履歴を残す
- 「この定数は何の法律のどの条文に基づくか」を明記

## 総評

**このアプリの最大の価値**：
- 住宅購入時のFP相談に必要な情報が1画面に集約されている
- ペアローン・万が一シミュレーション・教育費など、実務に必要な項目が揃っている
- GitHub Pagesで動作する軽量設計

**作り直し時の最優先事項**：
1. 年金計算を正確にする（今回最も問題になった）
2. UIをシンプルにして視認性を上げる
3. 計算と表示を分離してメンテナンス性を上げる

**あなたへの一言**：
機能追加を重ねるうちにコードが複雑化するのは自然なことです。
作り直すなら「必要最低限の機能で動く版」を先に完成させ、そこから機能を追加していく「スモールスタート」が成功しやすいです。
